Night Family Self-hosted orchestration for AI coding agents

While you sleep, your Night Family ships code. A self-hosted system that turns labeled GitHub issues into reviewed pull requests, run by a fleet of autonomous AI workers you control end-to-end.

A central Household orchestrates everything: it imports issues, dispatches work, and serves a live dashboard. Around it, any number of Members connect over WebSocket, each running its own LLM (Anthropic, Gemini, or OpenAI) inside a sandboxed container and acting under its own GitHub identity (one PAT per Member). Members clone the repo, write code, open draft PRs, and review each other. The final merge is always a human's call.

What it does

Issue → PR pipeline Label an issue night; a Member picks it up, branches, commits, and opens a draft PR.
Parallel review Multiple Members (and humans) review the same PR. Branch protection decides when it can merge.
Multi-provider Each Member runs one of Anthropic, Gemini, or OpenAI. Mix and match per role or budget.
You hold the keys Self-hosted. Each Member carries its own GitHub PAT — Household never holds your credentials. Members run in sandboxed containers on your VMs.

Workflow

The dashboard supports manual task entry, but the day-to-day flow is driven entirely by GitHub:

  1. You (or anyone with repo access) labels a GitHub issue night. Household receives the issues.labeled webhook and imports the issue as an implement task. Issues opened with the label already attached are imported on the opened event the same way.
  2. Household dispatches the task to an idle Member that matches its skills. The Member clones the repo using its own GitHub PAT (set locally as GITHUB_PAT) and creates a worktree on a branch named pr/night/<task-id-prefix>-<slug>.
  3. The Member's agent loop edits files, runs commands, and commits as Night Family <noreply+night@local> with a Co-Authored-By: Night <member-name> trailer.
  4. When the agent is done, the Member pushes the branch and opens a draft PR via gh pr create. The PR body ends with a 🤖 Authored by Night Family member footer linking back to the task.
  5. Household sees the pull_request.opened webhook, matches the PR to the task by branch name, and transitions the task to in-review. Other Members (and humans) can now review.
  6. On approval the task moves to awaiting-merge; on merge it transitions to done and a pr.merged notification fires. Branch protection rules decide who can actually merge — Members never self-merge.

Recommended branch protection

Branch protection isn't required for Night Family to track approvals — the verdict a reviewer Member emits is recorded internally — but it's how you keep humans in control of what actually merges. Suggested ruleset on the target branch (typically main):

  • Require pull request reviews before merging, with at least 1 approving review. GitHub forbids approving your own PR, so a Member can only satisfy this rule for a PR opened by a different GitHub account — i.e. another Member running under a different PAT, or a human reviewer.
  • Dismiss stale reviews on push — when a Member responds to feedback and pushes, the prior approval should not silently carry over.
  • Optionally require status checks (CI green) before merging. Members can run tests in their sandbox, but CI is the authoritative gate.

Solo single-PAT setups can't satisfy "1 approving review" — same-account reviews can only be of type Comment, never Approve. You have two clean options:

  • Run a second Member under a second GitHub account (e.g. a +alias Gmail address) and use it as a dedicated reviewer. Branch protection works fully.
  • Skip branch protection for solo setups and rely on Household's internal verdict — Member approvals still drive awaiting-merge, you just merge manually as the owner. Do not require approval if your only Member is the same GitHub account that opens PRs, or the loop deadlocks waiting for an approval that can never happen.

Conventions to know

  • The night label drives the lifecycle. Adding it imports / dispatches; removing it cancels the matching task; closing the issue cancels too (unless the task is already done or awaiting-merge, which the PR webhook owns). Cancellation reaches an in-flight Member as task.cancel.
  • Branch prefix pr/night/ is how Household matches PRs back to tasks. Don't rename agent-created branches if you want status updates to keep flowing.
  • Task kinds: implement, review, respond, summarize, estimate. Webhook-driven imports are always implement; the others are launched manually from the dashboard or as side-effects of PR review.
  • One Member, one provider. Each Member container runs a single LLM (Anthropic / Gemini / OpenAI) — running a fleet with mixed providers gives you an automatic second opinion during parallel review.

How it's built

  • TypeScript monorepo (shared · household · member)
  • Hono + Drizzle (SQLite) on the server, React + Vite for the dashboard
  • WebSocket protocol with reconnect-tolerant event replay
  • GitHub OAuth login, HMAC-validated webhooks, idempotent delivery dedup
