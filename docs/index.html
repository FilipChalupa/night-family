Night Family Self-hosted orchestration for AI coding agents

While you sleep, your Night Family ships code. A self-hosted system that turns labeled GitHub issues into reviewed pull requests, run by a fleet of autonomous AI workers you control end-to-end.

A central Household orchestrates everything: it imports issues, dispatches work, holds GitHub credentials, and serves a live dashboard. Around it, any number of Members connect over WebSocket, each running its own LLM (Anthropic, Gemini, or OpenAI) inside a sandboxed container. Members clone the repo, write code, open draft PRs, and review each other. The final merge is always a human's call.

What it does

Issue → PR pipeline Label an issue night; a Member picks it up, branches, commits, and opens a draft PR.
Parallel review Multiple Members (and humans) review the same PR. Branch protection decides when it can merge.
Multi-provider Each Member runs one of Anthropic, Gemini, or OpenAI. Mix and match per role or budget.
You hold the keys Self-hosted. Encrypted PATs in your DB. Members in sandboxed containers on your VMs.

Workflow

The dashboard supports manual task entry, but the day-to-day flow is driven entirely by GitHub:

  1. You (or anyone with repo access) labels a GitHub issue night. Household receives the issues.labeled webhook and imports the issue as an implement task. Issues opened with the label already attached are imported on the opened event the same way.
  2. Household dispatches the task to an idle Member that matches its skills. The Member clones the repo (using the per-repo PAT held by Household) and creates a worktree on a branch named pr/night/<task-id-prefix>-<slug>.
  3. The Member's agent loop edits files, runs commands, and commits as Night Family <noreply+night@local> with a Co-Authored-By: Night <member-name> trailer.
  4. When the agent is done, the Member pushes the branch and opens a draft PR via gh pr create. The PR body ends with a 🤖 Authored by Night Family member footer linking back to the task.
  5. Household sees the pull_request.opened webhook, matches the PR to the task by branch name, and transitions the task to in-review. Other Members (and humans) can now review.
  6. On approval the task moves to awaiting-merge; on merge it transitions to done and a pr.merged notification fires. Branch protection rules decide who can actually merge — Members never self-merge.

Conventions to know

  • The night label is the one and only trigger. Removing the label or closing the issue is a no-op; cancel the task from the dashboard instead.
  • Branch prefix pr/night/ is how Household matches PRs back to tasks. Don't rename agent-created branches if you want status updates to keep flowing.
  • Task kinds: implement, review, respond, summarize, estimate. Webhook-driven imports are always implement; the others are launched manually from the dashboard or as side-effects of PR review.
  • One Member, one provider. Each Member container runs a single LLM (Anthropic / Gemini / OpenAI) — running a fleet with mixed providers gives you an automatic second opinion during parallel review.

How it's built

  • TypeScript monorepo (shared · household · member)
  • Hono + Drizzle (SQLite) on the server, React + Vite for the dashboard
  • WebSocket protocol with reconnect-tolerant event replay
  • GitHub OAuth login, HMAC-validated webhooks, idempotent delivery dedup
